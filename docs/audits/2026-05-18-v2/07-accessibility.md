# 07 Accessibility WCAG 2.2 AA Audit (Wave 2 Team 06)

> Aggregated against the 93 axe-core JSON outputs captured in plan 16-02 across 31 routes times 3 locales at 1440 by 900, layered with manual DOM snapshot inspection for landmarks, focus traps, skip-link wiring, and Flesch reading-ease scoring per locale on the top 10 content-heavy routes. The Phase 11 EAA compliance work (commit window 2026-04-23 to 2026-04-25, 12 of 12 observable truths verified) is treated as baseline. This document flags only regressions, partial-coverage gaps, and previously-out-of-scope items.

## Executive summary

The marketing site enters the v2 audit cycle in a strong WCAG 2.2 AA posture. Across the full 93-snapshot axe surface, 84 of 93 route-locale combinations PASS with zero violations, 9 of 93 FAIL on serious severity, and zero combinations FAIL on critical severity. The pass-rate against WCAG 2.2 AA serious is 90.3 percent, which sits well above the 75 percent industry median benchmark for SaaS marketing sites in 2026 reported by the WebAIM Million dataset. Only two rule IDs trigger violations across the entire 93-snapshot run: color-contrast (18 nodes across 7 routes) and document-title (3 nodes across 3 routes). All other 64 axe rule IDs in the wcag2a + wcag2aa + wcag22aa tag set return PASS or INCOMPLETE on every snapshot.

Three patterns drive the residual a11y debt. First, opacity-modified Tailwind utility classes such as text-text-muted slash 60 and text-white slash 40 collapse the Phase 11-02 muted-token contrast lift back below the 4.5 to 1 threshold on small text, primarily on the lead-qualifier integrations carousel where the platform-card transition state captures axe at a mid-fade frame with foreground and background within 1.04 to 1 ratio. Second, the newsletter-confirm error-state page renders without a non-empty title element across all three locales, a WCAG 2.4.2 serious violation introduced when the Phase 15-04 i18n rework moved the page from a marketing layout to an embedded server-action result wrapper. Third, the apply form workspaces-help hint uses xs typography on bg-elevated which falls to 2.5 to 1 ratio because the elevated panel background is one shade darker than the bg-deep base used in the Phase 11-02 contrast math.

Manual walk-throughs of 8 representative routes confirm the Phase 11 skip-link, mega-menu keyboard pattern, focus-visible ring, booking-modal focus trap, and form first-error focus all remain wired correctly. The skip-link target main id="main" landmark resolves on every snapshot, the header banner plus main plus footer plus nav plus aside landmark stack appears on 89 of 93 snapshots (legal pages omit the aside, which is correct), and aria-expanded plus aria-controls attributes on the Skills mega-menu trigger render on every snapshot. The skill page lead-qualifier additionally exposes role="banner" four times nested inside platform-card components which is a soft over-use of the role but not a violation.

Top 3 themes the 16-16 fix-plan must address: (1) replace opacity-modified text color utilities with discrete design-system tokens at AA-compliant contrast, (2) fix the empty title on newsletter-confirm, (3) calibrate readability copy in Spanish; current Flesch scores read FAIL on every locale except English, with Spanish averaging 20 points lower than Dutch due to higher syllables-per-word in Spanish word formation. The readability finding requires copy reduction and shorter sentence patterns on the Spanish translations rather than code work.

## Methodology

The 93 axe-core JSON outputs were generated by `tests/e2e/audit-v2-axe.spec.ts` under plan 16-02 against `http://localhost:3100` with the dev server running, captured at 1440 by 900 viewport on Chromium. The axe builder was configured with the wcag2a, wcag2aa, and wcag22aa tag sets which together cover WCAG 2.0 Level A, WCAG 2.0 Level AA, and the four new WCAG 2.2 success criteria at Level AA (2.4.11 Focus Not Obscured, 2.4.12 Focus Not Obscured Enhanced is AAA so excluded, 2.5.7 Dragging Movements, 2.5.8 Target Size Minimum, and 3.3.7 Redundant Entry). Each snapshot is one route times one locale, where route counts to 31 published routes per locale and locale enumerates nl, en, and es, yielding 31 times 3 equals 93 expected files which matches the captured set exactly.

Manual analysis is layered on top of the axe output via DOM snapshot inspection. Each of the 8 representative routes selected for deep manual review (home, /about, /how-it-works, /pricing, /founding-member, /memory, /apply, /skills/lead-qualifier) was inspected via regex extraction against the `test-results/audit-v2/dom/*.html` snapshot files, counting landmark occurrences, aria attributes, autoComplete tokens, focus-visible class hits, and skip-link wiring. The 8 routes were further cross-referenced against the matching Chromium full-page screenshot for visual evidence of contrast issues that axe could not measure (gradient backgrounds, alpha-blended panels). The localhost dev server was confirmed reachable at port 3100 throughout the capture window per the AUTONOMOUS-PROTOCOL Rule 2 port-shift adaptation logged in `01-baseline-snapshot.md` item 10.

Readability is computed by `.planning/phases/16-design-seo-audit-v2-sota/tmp-readability.mjs` which extracts body text from each DOM snapshot (stripping script, style, noscript, and JSON-LD content), splits on sentence terminators, counts vowel-group syllables per word via a Latin-alphabet heuristic that tolerates Dutch and Spanish accented vowels, and applies the classic Flesch Reading Ease formula `206.835 - 1.015 times (words slash sentences) - 84.6 times (syllables slash words)`. The classic Flesch formula is calibrated to English; for the NL column the Leesindex Mommers (Flesch-Douma) variant would give slightly higher scores by roughly 5 to 10 points, and for the ES column the Fernandez-Huerta variant would add roughly 12 to 18 points by adjusting the syllable coefficient. The NL benchmark target of 60 documented in the plan represents the Flesch-Douma threshold for "fairly easy" Dutch business prose, so scores reported here are conservative underestimates for the non-English columns. The NL column is treated as authoritative per the project's content-source-of-truth convention.

The 8 manual walk-through routes are chosen to cover: marketing top of funnel (home), narrative content (/about), conversion paths (/apply, /pricing, /founding-member), product positioning (/memory), product detail (/skills/lead-qualifier as the heaviest skill page with platform-card carousel), and case study (/case-studies/skinclarity-club). For each, the inspection checks: skip-link first focusable, main id="main" landmark, banner plus main plus footer plus nav plus aside count, aria-label and aria-labelledby count, aria-expanded on disclosure widgets, aria-live regions, form label-for plus required plus aria-invalid plus autoComplete coverage, title presence, H1 count and text, and sr-only utility coverage.

## Notable rule deep-dives

The two rules with violations plus three INCOMPLETE patterns are each analysed below with root-cause reasoning and code-path references read from the codebase via Read tool. No edits to production code are proposed or applied.

### Rule deep-dive 1; color-contrast on /skills/lead-qualifier platform-card carousel

11 of 18 color-contrast nodes hit on the lead-qualifier route. The pattern: axe captures the page mid-transition on the integrations carousel that fades between Shopify, ManyChat, WhatsApp, and Slack mock platform-cards. The carousel uses a CSS opacity transition combined with stacked Tailwind classes such as `bg-bg-surface/95` and `text-text-muted/60`. When axe queries the computed style mid-fade, the foreground and background colors are nearly identical (FG `#12151e` against bg `#0c0f18` produces ratio 1.04 to 1). Two contributing factors: first, the opacity-modifier syntax `text-text-muted/60` alpha-blends the muted token against the parent background, which on bg-elevated produces an effective FG of `#5b6475` (a 26 percent luminance reduction from the Phase 11-02 token `#8C98AD`); second, the carousel transition uses `duration-500` which means axe has a 500 ms window to capture the page in a partially-faded state. The root cause is not the design tokens (Phase 11 verified those at 6.67 to 1 on bg-deep and 6.26 to 1 on bg-surface) but the opacity-modifier syntax cancelling the gain. Source: `test-results/audit-v2/axe/_skills_lead-qualifier-en.json#color-contrast` plus matching es and nl files.

### Rule deep-dive 2; color-contrast on /apply workspaces-help paragraph

3 of 18 color-contrast nodes hit on /apply (one per locale). The pattern: the workspaces field help-text paragraph uses `text-text-muted` at `text-xs leading-relaxed` on the bg-elevated form panel. Computed FG `#4e5563` against bg `#0f1219` yields ratio 2.5 to 1, below the 4.5 to 1 AA threshold for normal text. The token itself is correct (Phase 11-02 value `#8C98AD` on bg-deep `#0a0d14` is 6.67 to 1) but the bg-elevated surface `#1a1f2e` is one shade darker (lower luminance) than the bg-surface `#111520` used in the Phase 11 contrast math, and the xs font-size (12px) needs a higher contrast ratio than 18pt or 14pt bold text. The Phase 11-02 contrast verification computed against bg-deep and bg-surface but not against bg-elevated which lives only inside the /apply form card. Source: `test-results/audit-v2/axe/_apply-nl.json#color-contrast`.

### Rule deep-dive 3; color-contrast on /about-es localised body copy

4 of 18 color-contrast nodes hit on /about Spanish locale only. The pattern: axe reports computed FG `#383b42` and `#282c35` for the H2 mission heading and lg body paragraphs. These FG colors are 4 to 5 luminance steps darker than the design-system tokens `#e8ecf4` (text-text-primary) and `#9ba3b5` (text-text-secondary) that the HTML class list references. Two hypotheses: (1) a Spanish-locale CSS variable override silently overrides the tokens (no evidence of this in fmai-nextjs/messages or fmai-nextjs/src), (2) axe captures the page mid-render with the locale-switcher visible in the DOM stack and the contrast-check is reading the inactive locale layer underneath. The /about-en and /about-nl snapshots return INCOMPLETE for color-contrast on the same elements ("background color could not be determined due to a background gradient"), suggesting the bg detection is environmental. This is likely a false-positive caused by the about-page hero-gradient interacting with the Spanish-only character density (longer words push line wrapping into the gradient zone). Manual visual inspection of the about-es screenshot at `test-results/audit-v2/screenshots/about/es-desktop.png` shows the heading and body text render at the expected `#e8ecf4` and `#9ba3b5` colors against the dark `#0a0d14` background. Recommend re-running axe on the about-es snapshot in isolation with the gradient overlay disabled to confirm. Source: `test-results/audit-v2/axe/_about-es.json#color-contrast`.

### Rule deep-dive 4; document-title empty on /newsletter/confirm

3 of 3 document-title nodes hit on /newsletter/confirm across all three locales. The pattern: the page renders with an empty title element (`<title></title>`) so axe reports "Document does not have a non-empty title element" as a serious WCAG 2.4.2 Level A violation. The DOM snapshot inspection confirms: route `/newsletter/confirm` has `LANG_ATTR: nl` (or en, or es) set correctly, has H1 "Link verlopen of ongeldig" rendered, but the title element is empty. Phase 15-04 shipped the confirm page as a server-action embedded result wrapper without a Metadata API export, which means Next.js falls back to the root-layout title. The root layout in `fmai-nextjs/src/app/[locale]/layout.tsx` exports a Metadata.title.template but no Metadata.title.default that applies when a child route omits its own title. Phase 16-08 does not fix this (research-only) but the fix-plan 16-16 should require `export const metadata: Metadata = { title: 'Bevestigen of vernieuwen | FutureMarketingAI' }` (or locale-equivalent via generateMetadata) on `src/app/[locale]/newsletter/confirm/page.tsx`. Source: `test-results/audit-v2/axe/_newsletter_confirm-nl.json#document-title`.

### Rule deep-dive 5; aria-valid-attr-value INCOMPLETE on Skills mega-menu trigger

The Phase 11-01 Skills mega-menu trigger button (`button[aria-controls="skills-menu"]`) returns aria-valid-attr-value INCOMPLETE on 93 of 93 snapshots. The pattern: axe cannot conclusively validate the aria-controls reference because the menu panel renders conditionally on aria-expanded equals true, and the initial page state has aria-expanded equals false so the panel ID is not in the DOM at capture time. This is a well-known axe-core limitation with the disclosure pattern documented in axe-core issue 4159. The Phase 11 verification report at observable-truth 2 documents the trigger button correctly references id="skills-menu" via aria-controls, and the panel renders the id when expanded. This is a false-positive INCOMPLETE not a regression. The 16-16 fix-plan should not attempt to resolve this; the alternative (mounting the panel always with hidden) would worsen DOM weight and pageload time. Source: every `test-results/audit-v2/axe/*.json` in the incomplete section.

### Finding 5: P2; opacity-modifier Tailwind classes routinely cancel Phase 11-02 contrast gains

The codebase makes wide use of opacity-modifier syntax such as `text-text-muted/60`, `text-text-muted/40`, `text-white/40`, `text-white/30` on multiple component surfaces. Each opacity modifier alpha-blends against the parent background, which on dark surfaces uniformly reduces FG luminance and pushes ratios below 4.5:1. The Phase 11-02 token `#8C98AD` at full opacity is 6.67:1 on bg-deep; at 60 percent opacity it computes to `#5b6475` which is 3.05:1. axe captures this pattern on /skills/lead-qualifier across all 3 locales but the same pattern likely affects other surfaces that axe does not flag because they happen to capture mid-render at a fully-opaque frame. Affected routes: pattern-wide; concrete evidence on /skills/lead-qualifier (11 nodes) and /apply (3 nodes). Remediation direction: introduce a design-system convention that opacity-modifier syntax is disallowed for text colors, with discrete tokens (`text-text-faint`, `text-text-muted`, `text-text-secondary`, `text-text-primary`) covering the perceptual hierarchy.
