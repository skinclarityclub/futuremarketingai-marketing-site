    Wire the existing CalendlyModal component and useCalendlyBooking hook into the three locations that need Calendly booking functionality.

    **1. Layout (fmai-nextjs/src/app/[locale]/layout.tsx):**

    The CalendlyModal needs a global client-island wrapper in the layout so any component can trigger it. Create a new client component `CalendlyIsland` at `fmai-nextjs/src/components/interactive/CalendlyIsland.tsx`:

    ```tsx
    'use client'

    import { CalendlyModal } from './CalendlyModal'
    import { useCalendlyBooking } from '@/hooks/useCalendlyBooking'

    export function CalendlyIsland() {
      const { isOpen, prefill, url, closeCalendly } = useCalendlyBooking()
      return <CalendlyModal isOpen={isOpen} onClose={closeCalendly} url={url} prefill={prefill} />
    }
    ```

    Then in `layout.tsx`, import and render `<CalendlyIsland />` alongside `<ChatWidgetIsland />` and `<CookieConsentBanner />`.

    BUT WAIT -- useCalendlyBooking uses local useState, so a layout-level CalendlyIsland and ProgressiveCTA/DemoOrchestrator would have separate state. Instead, the Calendly open/close state needs to be SHARED. The simplest approach: add `calendlyOpen`, `calendlyPrefill`, `openCalendly`, and `closeCalendly` to the existing `chatbotStore` (which is already a global Zustand store available everywhere).

    **Revised approach:**

    a. **Extend chatbotStore** (`fmai-nextjs/src/stores/chatbotStore.ts`): Add to the store interface and implementation:
       - `calendlyOpen: boolean` (default: false)
       - `calendlyPrefill: { name?: string; email?: string } | undefined` (default: undefined)
       - `openCalendly: (prefill?: { name?: string; email?: string }) => void`
       - `closeCalendly: () => void`

    b. **Create CalendlyIsland** (`fmai-nextjs/src/components/interactive/CalendlyIsland.tsx`):
       ```tsx
       'use client'
       import { CalendlyModal } from './CalendlyModal'
       import { useChatbotStore } from '@/stores/chatbotStore'

       export function CalendlyIsland() {
         const { calendlyOpen, calendlyPrefill, closeCalendly } = useChatbotStore()
         const url = 'https://calendly.com/futureai/strategy-call?background_color=111520&text_color=e8ecf4&primary_color=00D4FF'
         return <CalendlyModal isOpen={calendlyOpen} onClose={closeCalendly} url={url} prefill={calendlyPrefill} />
       }
       ```
       Import the Calendly URL from `@/config/calendlyConfig` (the `calendlyConfig.url` export).

    c. **Add CalendlyIsland to layout.tsx:** Import and render `<CalendlyIsland />` inside the Providers wrapper, after ChatWidgetIsland.

    **2. ProgressiveCTA (fmai-nextjs/src/components/chatbot/ProgressiveCTA.tsx):**

    Replace the two `<a href="https://calendly.com/...">` direct links with button elements that call `useChatbotStore().openCalendly()`:

    - Import `useChatbotStore` from `@/stores/chatbotStore`
    - Destructure `openCalendly` from the store
    - Replace the gate banner (15+ messages) `<a>` with `<button onClick={() => openCalendly()}>Book a Discovery Call</button>` (keep all existing Tailwind classes, change tag from `a` to `button`, remove href/target/rel)
    - Replace the strong banner (10-14 messages) `<a>` with `<button onClick={() => openCalendly()}>Schedule a Demo</button>` (same approach)
    - Keep the subtle banner (5-9 messages) as-is -- it links to /contact, not Calendly

    **3. DemoOrchestrator (fmai-nextjs/src/components/chatbot/demo/DemoOrchestrator.tsx):**

    The `handleCompletionBookCall` callback currently sends a chat message (`onSendMessage(BOOKING_STEP.userMessage)`). Change it to open the CalendlyModal instead:

    - Import `useChatbotStore` (already imported for other state -- just add `openCalendly` to the destructured fields)
    - Change `handleCompletionBookCall` from `onSendMessage(BOOKING_STEP.userMessage)` to `openCalendly()`
    - This makes the "Book a call" button on the DemoCompletionCard open the Calendly modal directly

    The useCalendlyBooking hook at `fmai-nextjs/src/hooks/useCalendlyBooking.ts` becomes UNUSED after this refactor (we moved state to chatbotStore for global access). Leave it in place -- it can serve as a standalone hook for future use cases where local state is sufficient.
