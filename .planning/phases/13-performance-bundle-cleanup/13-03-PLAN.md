  <files>fmai-nextjs/src/components/chatbot/tool-results/ProductCard.tsx, fmai-nextjs/src/components/chatbot/tool-results/TicketCard.tsx, fmai-nextjs/src/components/chatbot/tool-results/BookingCard.tsx, fmai-nextjs/src/components/chatbot/tool-results/KBArticleCard.tsx</files>

Audit 08 section 4 flags `ProductCard.tsx`, `TicketCard.tsx`, `BookingCard.tsx`, `KBArticleCard.tsx` as "pure render-from-props" that can be RSC. They live under `src/components/chatbot/tool-results/`.

Checklist per file before dropping `'use client'`:

1. No `useState`, `useEffect`, `useMemo`, `useCallback`, `useRef` imports from React.
2. No event handlers (`onClick`, `onChange`, etc.) that call custom JS. Static `<Link>` hrefs are fine; inline `onClick` is not.
3. No `motion/react` or `AnimatePresence` imports.
4. No hooks like `useTranslations` â€” if present, either keep as client OR rewrite to accept translated strings as props from the parent.
5. No `document.*` or `window.*` references.

If all checks pass: remove the `'use client'` directive at the top. The component still works when rendered from a client parent (the client parent's render tree can include server components via props-passing; but NOT via direct import â€” only the intro `'use client'` boundary matters). Since the parent `index.tsx` maps `ComponentType<{ data: any }>` and renders these cards, the parent stays client; the cards themselves become server-able BUT will still be treated as client in the current import graph.

**Practical note**: in App Router, client components can render child components. If a client parent imports a server component directly, the server component is actually executed as a client component. To get real RSC benefit, the tool-result cards would need to be rendered from a server component (e.g., a streaming RSC under a Suspense boundary from the chatbot response).

Given that the entire chat subtree is client (ChatWidget), demoting `'use client'` from these four files is mostly lint cleanup â€” it DOES NOT reduce the client bundle because they are reachable from a client ancestor.

**Revised scope for Task 10**: instead of demoting, apply the 14 `<a href="/contact/">` to `<Link>` fix on these four + `ServiceCard.tsx` + `LeadScoreCard.tsx` + `CaseStudyCard.tsx` + `ProgressiveCTA.tsx` + `ChatWidget.tsx`. This is per audit 08 rank 4 â€” i18n breakage via hardcoded locale-less hrefs.

```tsx
// before
<a href="/contact/">Contact</a>

// after
import { Link } from '@/i18n/navigation'
// ...
<Link href="/contact">Contact</Link>
```

If the Link import conflicts with something already there (some files import `Link from 'next/link'` raw, which strips the locale prefix), switch to `@/i18n/navigation`'s `Link`. Verify with a click from the chatbot in NL: the locale prefix `/nl` must survive the click.

This fix:
- Removes 14 lint errors (`@next/next/no-html-link-for-pages`) from the next build.
