<task type="auto">
  <name>Task 2: Create tiny pick() helper (avoid lodash dependency)</name>
  <files>fmai-nextjs/src/lib/i18n-pick.ts</files>
  <action>
Implement a minimal typed pick utility. Do NOT add `lodash` (or `lodash-es`) as a dependency; that is a 70 KB ghost for a one-function need.

```ts
// src/lib/i18n-pick.ts

/**
 * pick — extract a subset of top-level keys from an object.
 * Used to scope next-intl message namespaces passed to NextIntlClientProvider.
 *
 * Usage:
 *   const messages = await getMessages()
 *   <NextIntlClientProvider messages={pick(messages, ['common', 'nav'])}>
 */
export function pick<T extends Record<string, unknown>, K extends keyof T>(
  obj: T,
  keys: readonly K[]
): Pick<T, K> {
  const out = {} as Pick<T, K>
  for (const key of keys) {
    if (key in obj) {
      out[key] = obj[key]
    }
  }
  return out
}
```

Placement: `src/lib/i18n-pick.ts`. One named export. Fully typed. Zero dependencies. ~200 bytes of runtime code.

<task type="auto">
  <name>Task 4: Create scoped NextIntlClientProvider for the (skills) route group</name>
  <files>fmai-nextjs/src/app/[locale]/(skills)/layout.tsx</files>
  <action>
Check if `src/app/[locale]/(skills)/layout.tsx` exists. If not, create it. If it exists, extend it.

The skills route group has 12 skill pages, each with its own namespace like `skills-social-media`, `skills-voice-agent`, etc. If any skill page uses a client component that calls `useTranslations('skills-voice-agent')`, that namespace must be available inside a client provider wrapping the page.

Approach: wrap the entire `(skills)` subtree with a scoped provider that includes every `skills-*` namespace plus `skills-common` (if it exists) plus any other shared skill-level namespace identified in Task 1.

```tsx
// src/app/[locale]/(skills)/layout.tsx
import { NextIntlClientProvider } from 'next-intl'
import { getMessages } from 'next-intl/server'
import { pick } from '@/lib/i18n-pick'
import type { ReactNode } from 'react'

// All skill-namespaces. Use runtime filter to avoid hardcoding drift.
function getSkillsNamespaces(messages: Record<string, unknown>): readonly string[] {
  return Object.keys(messages).filter(
    k => k === 'skills-common' || k.startsWith('skills-')
  )
}

export default async function SkillsLayout({
  children,
  params,
}: {
  children: ReactNode
  params: Promise<{ locale: string }>
}) {
  const { locale } = await params
  const messages = await getMessages()
  const namespaces = getSkillsNamespaces(messages) as readonly (keyof typeof messages)[]

  return (
    <NextIntlClientProvider messages={pick(messages, namespaces)} locale={locale}>
      {children}
    </NextIntlClientProvider>
  )
}
```

Notes:
- next-intl supports nested providers. The inner one REPLACES the outer subset for its subtree. Therefore, keep `common`, `nav`, and any other globally-used-in-Skills-layout namespaces also in the pick list. Practical fix: the pick list for the skills layout is `[...GLOBAL_CLIENT_NAMESPACES, ...skillsNamespaces]`. Refactor the constant into `src/lib/i18n-namespaces.ts` if you prefer a shared export.
- Verify by running `npm run build` and `npm run dev`, then loading `/nl/skills/voice-agent`. If a client component throws "namespace not found", add that namespace.

If the skills pages are entirely server-rendered (no client useTranslations), this scoped layout is not needed. Check Task 1 inventory before creating this file. Skip this task if the inventory confirms all skills rendering is server-side.

Most likely path: `SkillPageTemplate.tsx` is a client component that pulls `skills-common` for shared tab/section labels. Confirm by reading the file.
